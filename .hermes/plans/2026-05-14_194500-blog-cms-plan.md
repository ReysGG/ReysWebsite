# Blog Website + Admin CMS Implementation Plan

## Goal
Build a scalable, secure, SEO-friendly blog system for BuildWebsite with:
- Public blog listing inspired by the Portfolio visual style.
- SEO-friendly article detail pages with clean keyword slugs.
- Pagination, filtering, search, category/year/tag filters.
- WordPress-like admin create/edit experience with rich content, images, formatting, draft/publish flow.
- Strong page-level security and robust error handling.
- Feature components split cleanly for easier scaling.

## Current Context

```prisma
model Post {
  id         String    @id @default(cuid())
  title      String
  slug       String    @unique

  excerpt    String?

  tags       String[]
```

## Security Requirements

### 1. Admin route access
All admin blog pages must be protected.

Options depending existing auth:
- If admin layout already protects `/admin`, verify it covers `/admin/blog`.
- If not, add a route-level guard.

Recommended helper:
`src/features/admin/lib/auth.ts`

Functions:
- `requireAdmin()`
- `getCurrentAdminUser()`

Every server action must call `requireAdmin()` even if the page is protected.
Do not trust UI-only protection.

### 2. Input validation
Validate all server action data:
- title required
- slug required and unique
- slug format: lowercase, hyphen, no unsafe chars
- content required for publish
- excerpt max length
- metaTitle recommended max 60 chars
- metaDesc recommended max 155-160 chars
- coverImage URL/path validation
- tags normalized and capped

Recommended constants:
- title max 120 chars
- slug max 90 chars
- excerpt max 240 chars
- metaTitle max 70 chars
- metaDesc max 170 chars
- tags max 10
- tag length max 32

### 3. Rich text / HTML safety
Because WordPress-like content allows styling and images, this is the biggest security risk.

Minimum requirements:
- Sanitize HTML before saving or before rendering.
- Allow only safe tags:
  - `p`, `br`, `strong`, `em`, `u`, `s`
  - `h2`, `h3`, `h4`
  - `ul`, `ol`, `li`
  - `blockquote`
  - `pre`, `code`
  - `a`
  - `img`
- Allow only safe attributes:
  - `href`, `target`, `rel` for links
  - `src`, `alt`, `title`, `width`, `height` for images
  - limited `class` only if needed
- Force external links to:
  - `rel="noopener noreferrer nofollow"`
- Block script/event attributes:
  - `script`
  - `iframe` initially, unless embed whitelist is implemented
  - `onerror`, `onclick`, etc.

If adding dependency is okay, use `sanitize-html` or `isomorphic-dompurify`.
If avoiding dependency, implement conservative stripping first, then improve.

### 4. Image safety
For images inserted in rich editor:
- Prefer URL input first, not raw uploads, unless upload handling is ready.
- Validate protocol: only `https`, `/local-path`, or trusted domain.
- Require alt text in admin UI.
- For external image domains, ensure `next.config` permits them before using `next/image`.
- Do not allow `javascript:` or data URLs by default.

### 5. Slug security
- Only allow `[a-z0-9-]`.
- Reject path traversal or slash.
- Unique constraint exists in Prisma.
- On collision, append suffix:
  - `slug`
  - `slug-2`
  - `slug-3`

### 6. Public page hardening
- Only render `published: true` posts on public detail/list.
- Draft should 404 on public route.
- Use `notFound()` for missing/unpublished post.
- Avoid leaking unpublished content through related posts or sitemap.

### 7. CSRF / Server Actions
Next Server Actions are generally protected, but still:
- Use POST-only server actions.
- Validate IDs exist.
- Re-check admin permission inside action.
- Do not expose delete/publish action without admin guard.

### 8. Error handling and logging
Each server action returns:

```ts
type BlogActionState = {
  success: boolean;
  message?: string;
  error?: string;
  fieldErrors?: Record<string, string>;
};
```

Internally log unexpected errors with `console.error`, but return safe generic messages to UI.

## Risks and Tradeoffs

### Rich text security
Risk: rendering editor HTML can cause XSS.
Mitigation: sanitize HTML and restrict allowed tags/attributes.

### ReactQuill SSR
Risk: editor can break server rendering.
Mitigation: isolate as client component and dynamic import with `ssr: false` if needed.

### Current schema lacks category/publishedAt
Tradeoff: phase 1 can use tags/date fallback. Proper category/publishedAt requires migration.

### Image upload not defined
Tradeoff: start with image URL input. Add upload/storage later.

### Draft preview
Public draft preview needs secure token/session design. Do not expose draft content publicly without auth.

## Open Questions

1. Should we add schema migration now for category, focusKeyword, featured, publishedAt, ogImage?
2. Should images be URL-only for now, or should we implement upload?
3. Should category be a free text field or fixed select options?
4. Should comments/likes be enabled publicly or hidden for now?
5. Should blog use Indonesian-only slugs/content, or support multilingual later?
